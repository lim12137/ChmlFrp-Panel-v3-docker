## 结果摘要
- 前端页面可正常打开，核心导航可达。
- `消息中心` 由 404 改为“暂无消息”空态。
- `隧道启用` 从原先 500/ENOENT 改为 503，错误原因明确。
- `停止隧道` 返回 200。
- `FRP` 状态页不再误判为运行中。

## 发现问题
1. **High**：本地启动隧道仍硬编码 `/app/frpc`、`/app/configs`，Windows 本地直接 `spawn /app/frpc ENOENT`。
2. **High**：FRP 日志与配置调试输出会写入 token / accessToken / refreshToken 等敏感值。
3. **Medium**：`/api/messages` 代理到上游 404，消息页直接报错。
4. **Medium**：隧道启用失败时前端只提示网络错误，原因不够明确。

## 根因
- FRP 管理器只按 Docker 路径初始化，没有区分本地 Windows 运行环境。
- 单隧道启动流程在进程失败前仍会继续生成状态，且日志输出未脱敏。
- 消息接口依赖上游返回，但上游当前没有该路径。

## 修复内容
- `backend/frp-manager.js`
  - 改为按运行环境解析 FRP binary / 配置 / 状态 / 日志路径。
  - 增加本地 binary 缺失检查，改为明确错误而不是 500 崩溃。
  - 给启动流程加了防重复 resolve/reject 保护。
  - 日志与配置调试输出统一脱敏。
- `backend/index.js`
  - `/api/messages` 在上游 404 时返回空消息列表。
  - `start-tunnel` 在缺少本机 FRP binary 时返回 503。
  - `frp/status` / 恢复状态改用实际运行目录。
- `frontend/src/components/TunnelManagement.js`
  - 启用失败时直接展示后端返回原因。

## 复验结果
- `POST /api/frp/start-tunnel`：`503`，返回明确原因：本机缺少 `frpc.exe`。
- `POST /api/frp/stop-tunnel`：`200`。
- `GET /api/messages`：`200`，空态正常。
- `GET /api/frp/status`：`isRunning=false`，`configPath` 指向本地运行目录。

# 隧道操作按钮状态修复复验

## 测试范围
- `TunnelManagement` 操作列启动/停用按钮。
- 状态来源：`/api/frp/status`、`/api/frp/running-tunnels`、`/api/tunnel`。

## 测试命令 / 请求
- `Invoke-RestMethod http://localhost:3001/api/frp/status`
- `Invoke-WebRequest http://localhost:3001/api/frp/running-tunnels -SkipHttpErrorCheck`
- `Invoke-RestMethod http://localhost:3001/api/tunnel`
- `npm run build`
- Chrome DevTools Protocol 脚本打开 `http://localhost:3000/dashboard/tunnels`，读取表格行按钮文案与计算后颜色。
- Chrome DevTools Protocol 脚本点击第一条“停用”，再次读取表格行按钮文案与计算后颜色。

## 接口结构确认
- `/api/frp/status` 返回 `data.activeTunnels: []`、`data.tunnelCount: 0`、`data.isRunning: false`，本地 FRP 管理器当前没有活跃进程记录。
- `/api/frp/running-tunnels` 当前返回 `404`，响应为 `{"code":-1,"state":"error","msg":"接口不存在","data":null}`。
- `/api/tunnel` 返回的隧道列表包含真实运行字段 `state`，本次复验中 4 条隧道均为 `state: "true"`、`nodestate: "online"`。

## 根因
- 前端操作列只使用 `/api/frp/status.data.activeTunnels` 生成 `activeTunnelIds`。
- 当前真实运行状态来自 `/api/tunnel` 的 `state: "true"`，而 `/api/frp/status.activeTunnels` 为空，导致运行中的隧道仍按“未运行”渲染绿色“启用/启动”按钮。
- 另有潜在类型问题：运行集合里的 `tunnelId` 可能是字符串，而隧道列表 `id` 是数字，直接 `Set.has(record.id)` 会匹配失败。

## 修复内容
- `frontend/src/components/TunnelManagement.js`
  - 统一隧道 ID 为字符串后比较。
  - 操作列和状态列同时合并本地 FRP 活跃集合与隧道列表 `state: "true"`。
  - 点击停用成功后加入本地停用覆盖集合，立即显示绿色“启用”；重新启动成功或列表状态变为未启用后清理覆盖。
  - 活跃隧道计数改为使用同一运行判断逻辑。

## 复验结果
- 构建：`npm run build` 成功，只有项目既有 ESLint warning。
- 状态矩阵脚本：
  - 本地运行状态优先，数字列表 ID 可匹配字符串运行 ID：PASS。
  - 刚停止的隧道本地覆盖为未运行：PASS。
  - `state: "true"` 显示运行中：PASS。
  - `state: "false"` 显示未运行：PASS。
- 页面复验：
  - 进入 `http://localhost:3000/dashboard/tunnels` 后，4 条 `state: "true"` 隧道均显示“运行中”，操作按钮均为红色 `rgb(255, 77, 79)` 的“停用”。
  - 点击第一条“停用”后，该行变为“未启用”，操作按钮变为绿色 `rgb(82, 196, 26)` 的“启用”；其余运行中隧道仍保持红色“停用”。

## 剩余问题
- `/api/frp/running-tunnels` 未实现；当前修复未新增接口，仅使用已有 `/api/tunnel.state` 做最小前端修复。
- `/api/frp/status` 只反映本地 FRP 管理器进程状态，不代表上游隧道列表的真实 `state`。
